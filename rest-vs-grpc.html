REST vs gRPC 비교
REST API
HTTP/1.1 + JSON
데이터 형식 JSON (텍스트)
직렬화 느림 파싱 비용 큼
프로토콜 HTTP/1.1 (연결당 1요청)
타입 안정성 약함 런타임 에러
브라우저 지원 네이티브
gRPC
HTTP/2 + Protocol Buffers
데이터 형식 Protobuf (바이너리)
직렬화 10x 빠름 바이너리
프로토콜 HTTP/2 (멀티플렉싱)
타입 안정성 강함 컴파일 체크
브라우저 제한적 gRPC-Web 필요
언제 무엇을 쓸까?
REST가 적합한 경우
브라우저 - 서버 통신
외부 공개 API
간단한 CRUD 작업
디버깅 편의성 중요
gRPC가 적합한 경우
서버 - 서버 내부 통신
마이크로서비스 간 통신
실시간 스트리밍
고성능, 낮은 지연 필요
핵심: 외부는 REST, 내부는 gRPC
브라우저 → API Gateway (REST) → 마이크로서비스들 (gRPC)
코드 비교
REST (HTTP + JSON)
// 요청
POST /users HTTP/1.1
Content-Type: application/json

{
  "name": "John",
  "age": 30
}

// 텍스트 파싱 필요
gRPC (HTTP/2 + Protobuf)
// user.proto 정의
message User {
  string name = 1;
  int32 age = 2;
}

// 바이너리로 전송 (크기 작음)
